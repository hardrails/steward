# Installed support contract

Documentation can move ahead of an installed binary. A release label alone also
does not say whether a machine is a supported Executor, which agent runtime was
qualified, or which limitations still apply. Steward therefore carries a small,
offline support contract in `stewardctl`.

```console
stewardctl support matrix
```

The default output is written for an operator. It names the supported controller,
Executor, and workstation roles; the qualified agent runtime; the production
sandbox; and the known limits that most often change a deployment decision.

For automation, request deterministic JSON:

<!-- cli-flags: support matrix | -output -->
```console
stewardctl support matrix -output json
```

The JSON has schema `steward.support-matrix.v1` and includes:

- the exact version reported by the binary and whether it is a stable or
  development/prerelease build;
- supported host roles, architectures, status, and prerequisites;
- qualified and explicitly unsupported agent runtimes;
- the production isolation and tenant-boundary contract;
- supported operator interfaces and authority modes;
- disconnected capabilities;
- the node manifest, state-format inspection command, and backup schema used at
  upgrade and recovery boundaries; and
- known limits that must not be mistaken for supported behavior.

The contract is intentionally closed. Unknown fields are not extension points,
and a runtime does not become supported merely because its container starts.
Support requires a fixed adapter contract, a documented workflow, enforced
failure behavior, and qualification evidence.

## Release asset

Each release publishes `steward-support_<version>.json`. The file is generated by
the stamped release binary and covered by `checksums.txt`. Before publication, CI
extracts the Linux release archive, asks its `stewardctl` for the JSON contract,
and requires a byte-for-byte match with the release asset.

```console
sha256sum -c checksums.txt
stewardctl support matrix -output json > installed-support.json
cmp installed-support.json steward-support_<version>.json
```

On macOS, use `shasum -a 256 -c checksums.txt` for the checksum step.

The support file is evidence of what the binary claims, not an independent
signature or a substitute for host qualification. Obtain `checksums.txt` through
a trusted channel, and run packaged preflight plus workload acceptance on the
exact kernel, systemd, Docker, gVisor, and architecture used in production.

## Upgrade use

The support contract points to, but does not replace, the per-node `release.json`
format ranges. Before changing a node release, inspect its retained state against
the target manifest:

```console
stewardctl upgrade inspect-formats \
  -signed-admission configured \
  -release-manifest /opt/steward/releases/<target-version>/release.json
```

An upgrade is supported only when the target manifest accepts every retained
format reported by the inspection. The support contract never turns an
incompatible state transition into an allowed one.
